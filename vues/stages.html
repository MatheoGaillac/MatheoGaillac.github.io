Situation

Bouton Switch

Toujours dans une réflexion d'optimisation de l'application web, nous avons remarqué que lorsque l'on souhaitait afficher plusieurs diaporamas en même temps, le site était drastiquement ralenti voire se stoppait. Nous avons donc décidé d'empêcher l'affichage en même temps de plusieurs diaporamas, en permettant à l'utilisateur de ne pouvoir choisir qu'un seul diaporama, les autres se désactivant automatiquement.

@app.callback(
	Output("switch_diaporama_meteo_2", "on"),
	Output("switch_diaporama_alertes_2", "on"),
	Output("switch_transversalite_2", "on"),
	Input("switch_diaporama_meteo_2", "on"),
	Input("switch_diaporama_alertes_2", "on"),
	Input("switch_transversalite_2", "on"),)
	def update_switch(meteo_on, alerte_on, transversalite_on):
	ctx = dash.callback_context
	button_id = ctx.triggered[0]['prop_id'].split('.')[0]
	if button_id == "switch_diaporama_meteo_2":
	  if meteo_on:
		return True, False, False
	  else:
		return False, False, False
	elif button_id == "switch_diaporama_alertes_2":
	  if alerte_on:
		return False, True, False
	  else:
		return False, False, False
	elif button_id == "switch_transversalite_2":
	  if transversalite_on:
		return False, False, True
	  else:
		return False, False, False
	else:
	  return False, False, False

Le code suivant représente une callback mise en place pour la gestion des boutons "switch". Tout se base sur des conditions booléennes, désactivant les autres boutons lorsque l'un d'eux est activé. Cela permet la surcharge de requête sur la base de données et sur la page. En somme, nous appelons une bibliothèque (ctx) pour obtenir le contexte du callback en cours d'exécution, cela sert à déterminer l'élément déclencheur du callback. Par la suite, nous accédons au premier élément de la liste, en renvoyant l'id de la propriété pour finalement initialiser une variable contenant l'id du bouton cliqué. Il y a ensuite une vérification du bouton déclenché avec la gestion en booléen de l'activation ou non, pour n'en garder qu'un.

Situation

Optimisation des ressources allouées

L'un des problèmes majeurs lié à l'optimisation et les performances de l'application web se trouvait finalement dans la gestion des ressources allouées (CPU et RAM). C'est pourquoi j'ai décidé de mettre en place un tableur Excel, exposant les ressources à dispositions, et les configurations. Le cumul est aussi calculé afin d'éviter tout problème. Quand le max est affiché en vert, c'est que le "pod" a encore de la marge, en jaune quand il s'approche du maximum et en rouge quand il est au maximum possible ou au delà. Ce tableur nous a permis de prévoir la nouvelle répartition des ressources pour rendre le site plus performant.

Pour ce faire, j'ai dû éditer les fichiers de "deployment" qui permettent de construire les pods individuellement pour faire marcher l'application. Les ressources ont donc été modifiées en augmentant leurs capacités minimales et maximales !

Cependant, lors du premier push sur le repo et la mise en production, une erreur est survenu, expliquant qu'il n'arrivait pas à comprendre et donc compiler la partie "ressources". En effet, selon a typologie du langage actuel, "ressources" est censé ne prendre qu'un seul "s", hors ici il en avait 2, d'où l'erreur qui empêchait la compilation et l'interprétation des informations. Ce problème datait d'il y a plus d'un an maintenant, et puisque l'application n'avait pas de ressources en paramètres, elle prenait celle par défaut, ce qui bridait donc le site.

Finalement, comme observé sur le Dashboard ou pannel d'administration de l'application, depuis que le bug a été corrigé et les ressources augmentées, l'application web fonctionne bien mieux ! Le site ne plante plus, il est largement plus rapide et les retours des utilisateurs en est positif.
